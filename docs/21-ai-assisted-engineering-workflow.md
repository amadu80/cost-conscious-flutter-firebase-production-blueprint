# AI-assisted engineering workflow

This case study is not primarily about code generated by an AI assistant. It is about using AI as a structured engineering collaborator across discovery, implementation, review, operations, and learning—while keeping human accountability and evidence at the center.

## Problem

A solo engineer or small team must cover architecture, Flutter, Firebase, security, Terraform, releases, tests, observability, incidents, documentation, and cost. The breadth creates two risks:

- Important questions are missed because no specialist is present in every domain.
- AI output is accepted too quickly because it appears complete and confident.

The objective was therefore not “ask AI to build the application.” It was to create a repeatable loop in which AI accelerates investigation and production work, while the engineer owns intent, access, trade-offs, verification, and release decisions.

## Options considered

1. **Use AI only for code completion:** low process overhead, but wastes its ability to review systems, compare options, and document reasoning.
2. **Delegate whole features with minimal supervision:** fast when successful, but encourages hidden assumptions, broad changes, invented claims, and weak ownership.
3. **Use AI as a controlled engineering collaborator:** provide scoped context, demand alternatives and evidence, make surgical changes, run gates, challenge the result, and preserve attribution.

## Decision

Use the third approach. AI assistants may investigate, propose, edit, test, document, and critique within an explicitly defined scope. The human remains accountable for requirements, product and safety choices, credentials, production access, destructive actions, risk acceptance, and deployment.

## The working loop

### 1. Establish context before asking for change

The assistant reads relevant architecture rules, current files, recent or full Git history, incident reports, deployment guides, and known constraints. This reduces advice based on an imaginary greenfield project.

Useful prompt pattern:

> Read the relevant history and documents first. State the current design, constraints, and evidence. Do not change files yet.

### 2. Define the problem and authorization boundary

The human states the desired outcome, target files or systems, current error, release scope, and what must not change. Ambiguous structural or destructive work is confirmed before action.

Useful prompt pattern:

> Diagnose why the hosted web application fails during initialization. Focus on the web release; mobile remains deferred. Do not weaken backend enforcement as a workaround.

### 3. Require options, decision, and why

The assistant must identify credible alternatives—including doing nothing—then explain cost, security, operational effort, reversibility, and exit triggers. This prevents technology choices from becoming unexplained preferences.

Useful prompt pattern:

> Compare a custom API, direct Firebase access, and a backend-for-frontend. Recommend one for the current constraints, list the risks it introduces, and define when to revisit it.

### 4. Make surgical changes

Changes are limited to the approved outcome and matched to existing architecture and style. Unrelated refactoring and speculative abstractions are rejected. Generated configuration, infrastructure, rules, and client behavior are treated as one change when the runtime depends on all of them.

### 5. Verify proportionately to risk

AI does not mark work complete because a patch applied. Verification can include formatting, static analysis, deterministic tests, Rules allow/deny cases, Terraform validation and plan review, clean production builds, browser smoke tests, cache-upgrade tests, public headers, release identity, or restore exercises.

The human distinguishes:

- **Code evidence:** the implementation exists.
- **Test evidence:** a controlled assertion passes.
- **Deployment evidence:** the intended artifact is live.
- **Operational evidence:** failure is detected and recovered.

### 6. Challenge the output with another role

After implementation, the assistant is prompted to act as an adversarial architect, SRE, security reviewer, privacy reviewer, or cost reviewer. The goal is to identify unsupported claims, missing failure paths, and ways the solution could become more expensive or unsafe.

Useful prompt pattern:

> Assume this decision failed in production. Be explicit about what is unproven, how the control can fail, what evidence is missing, and which mitigation must happen first.

### 7. Preserve the decision and its limits

Documentation records the problem, alternatives, decision, rationale, consequences, verification, known gaps, mitigation, and revisit trigger. Incidents are converted into post-mortems; future failures are explored through pre-mortems.

### 8. Attribute AI participation

Project prompts are recorded in a sanitized history with timestamp, user command, assistant/engine, and response summary. 

**VCS Attribution Standards:**
When an assistant generates a Git commit, append a machine-readable trailer to the commit message:
`AI-Engine: <engine_name>`

This ensures that AI-authored changes can be identified via `git blame` and commit history audits, supporting long-term provenance and debugging without cluttering source-code comments.

## How different AI uses contributed

### Architecture and refactoring

AI helped inspect coupling, propose repository boundaries, DTOs, typed results, use cases, declarative routing, state handling, and technology alternatives. The important output was not the number of files generated; it was preserved reasoning and tests around boundaries.

### Testing and quality gates

AI helped identify untested states, create deterministic cases, stabilize brittle tests, enforce formatting and analysis, and distinguish unit, widget, emulator, integration, and hosted release evidence.

### Security hardening

AI helped audit Rules, App Check rollout, IAM, secrets, headers, CSP, OAuth configuration, data isolation, and abuse risks. Security changes were treated as layered controls rather than a single product setting.

### Infrastructure and cost

AI helped translate console-managed resources into Terraform, review provider limitations, remote state, regions, budgets, monitoring, storage lifecycle, TTL, and edge configuration. Cost was considered alongside operational effort and failure exposure.

### Release engineering and incidents

AI helped correlate build configuration, plugin registration, service workers, caches, hosting headers, public-origin behavior, and rollback. Pre-mortems and post-mortems converted incidents into release gates and known-gap registers.

### Documentation and self-review

AI helped turn commit history into a sanitized case study, but was also instructed to challenge its own output. Polished documentation can overstate maturity as easily as polished code.

## Guardrails

- Never provide production secrets, personal data, private tokens, or Terraform state to an assistant.
- Treat all generated code and configuration as untrusted until reviewed and tested.
- Require human approval for destructive, production, permission, billing, and external communication actions.
- Do not allow AI to invent test results, audits, compliance status, benchmarks, or deployed evidence.
- Keep security, privacy, safeguarding, and product decisions owned by accountable humans.
- Prefer small diffs and explicit targets; reject unrelated refactoring.
- Stop repeated build attempts when external dependencies are the blocker.
- Verify current provider and platform behavior using authoritative sources when it may have changed.
- Preserve user-authored changes and inspect dirty worktrees before editing.

## Common AI failure modes observed or anticipated

### Confident completion without runtime evidence

An assistant may see passing unit tests and describe the release as ready. Mitigation: define evidence levels and require public-origin or failure-exercise proof for operational claims.

### Solving symptoms by weakening controls

An initialization failure can tempt an assistant to disable App Check, CSP, Rules, or cache protections. Mitigation: preserve the security objective, diagnose the integration boundary, and use staged rollback only when risk is explicitly accepted.

### Broad refactoring

AI can turn a focused fix into a new architecture. Mitigation: surgical-change rules, target files, diff review, and a separate decision for structural work.

### Hallucinated APIs or provider support

Cloud and Flutter ecosystems change. Mitigation: inspect installed versions and official documentation, validate configuration, and record provider limitations.

### Test quantity mistaken for quality

AI can produce many tests that repeat implementation details or pass without exercising the failure. Mitigation: require behavioral assertions, negative paths, mutation of assumptions, and evidence that the test fails when the control is removed.

### Documentation inflation

AI can create comprehensive prose faster than evidence. Mitigation: claim classification, evidence links, and a rule to stop adding breadth when verification lags.

## Cost-conscious use of AI

AI reduces search, drafting, refactoring, and documentation time, but also consumes review time and can generate unnecessary scope. Measure its value through outcomes:

- Time from problem report to verified cause
- Escaped defect and recurrence rate
- Review effort and diff size
- Test and incident evidence quality
- Infrastructure and provider cost avoided or introduced
- Documentation kept accurate after change

Do not optimize for prompts sent, lines generated, commits produced, or tests counted. Those are activity measures.

## What this workflow does not prove

Using multiple assistants, recording prompts, and attributing commits does not prove correctness, originality, security, compliance, or production readiness. The evidence remains tests, reviewed changes, deployed artifact identity, operational exercises, user outcomes, and accountable human decisions.

The strongest use of AI in this project was not generating more code. It was repeatedly expanding the set of questions asked, accelerating bounded implementation, and forcing decisions and failures to become explicit. That value disappears if the human stops verifying the answers.
